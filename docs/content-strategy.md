# 內容策略

> AIDC.work 顧問內容資產庫的規劃與撰寫指南

## 1. 內容策略目標

AIDC.work 的內容不只是行銷文案，而是**可累積、可複用、可展示的顧問資產**。

| 目標 | 說明 |
|------|------|
| 建立信任 | 透過案例與實務文章，讓潛在客戶看見真實能力 |
| 提供價值 | 讀者即使尚未合作，也能從模板與文章獲得實用知識 |
| 展示成果 | 呈現 AI 開發、Agent 工作流與 Vibe Coding 的實際產出 |
| 支援轉換 | 引導讀者從內容消費走向課程報名或合作洽詢 |

## 2. 內容支柱（Content Pillars）

### 支柱一：企業 AI 導入與策略

- 企業 AI 成熟度評估
- 導入路線圖規劃
- 常見陷阱與解法
- 決策者溝通與內部推動

**目標讀者**：CXO、部門主管、數位轉型負責人

### 支柱二：AI 工具與 Vibe Coding

- AI 工具選型與團隊培訓
- Vibe Coding 方法與實戰技巧
- AI 輔助開發工作流程
- 從原型到可維護程式碼的轉換

**目標讀者**：開發者、技術主管、產品經理

### 支柱三：AI 系統與 Agent 工作流

- Agent 架構設計
- 工作流自動化實作
- 工具串接與整合模式
- 可靠性、安全性與可觀測性

**目標讀者**：架構師、資深工程師、IT 主管

### 支柱四：顧問模板與實務工具

- 評估檢核表
- 工作坊引導文件
- Prompt 與工作流模板
- 會議紀錄與決策框架

**目標讀者**：所有角色，特別是需要「拿回去就用」的團隊

## 3. 內容類型與格式

### 3.1 部落格文章（`content/blog/`）

| 子類型 | 篇幅 | 更新頻率（建議） |
|--------|------|------------------|
| 觀點文章 | 800–1,500 字 | 每月 1–2 篇 |
| 技術教學 | 1,500–3,000 字 | 每月 1 篇 |
| 導入心得 | 1,000–2,000 字 | 不定期 |

**必備 frontmatter 欄位**：

```yaml
title: "文章標題"
description: "SEO 與列表摘要"
date: "2026-06-05"
author: "Jack Y. J. Hwang"
tags: ["AI Agent", "企業導入"]
category: "策略" # 策略 | 技術 | 實務
draft: false
```

### 3.2 案例研究（`content/cases/`）

| 要素 | 說明 |
|------|------|
| 背景 | 客戶產業、規模、面臨的挑戰（可匿名處理） |
| 解法 | 顧問介入方式、技術選型、工作流設計 |
| 成果 | 量化或質化成效 |
| 可複製經驗 | 其他企業可參考的洞察 |

**撰寫原則**：

- 未取得授權時，使用匿名化名與產業描述
- 聚焦「問題 → 方法 → 成果」，避免過度推銷
- 可搭配架構圖、流程圖（Mermaid 或圖片）

### 3.3 課程與培訓（`content/courses/`）

| 要素 | 說明 |
|------|------|
| 學習目標 | 學員完成後能做什麼 |
| 課程大綱 | 模組與時數 |
| 適合對象 | 角色、先備知識 |
| 形式 | 實體 / 線上 / 企業內訓 |

### 3.4 模板與工具（`content/templates/`）

| 要素 | 說明 |
|------|------|
| 用途 | 解決什麼問題 |
| 使用方式 | 步驟說明 |
| 格式 | Markdown、Notion、PDF、Prompt 等 |
| 下載或複製 | 內嵌內容或外部連結 |

## 4. 語氣與風格

### 4.1 寫作原則

- **專業但平易近人**：像資深顧問在說明，不是推銷員在喊口號
- **務實導向**：每篇文章應有至少一個可帶走的觀點或做法
- **結構清晰**：使用標題、列表、表格，方便掃讀
- **避免流行語堆砌**：少用「顛覆」「革命性」等空泛形容

### 4.2 語言規範

- 正文使用**繁體中文**
- 技術名詞、工具名稱、專業職稱可保留英文（如 AI Agent、Vibe Coding、CTO）
- 中英混用時保持一致性，首次出現可附中文說明

### 4.3 標題撰寫

- 具體優於抽象：「三個常見的 AI Agent 整合錯誤」優於「AI Agent 最佳實務」
- 承諾讀者可獲得的價值
- 長度建議 15–30 字

## 5. 內容發布流程

```text
1. 選題（對應內容支柱與使用者需求）
2. 撰寫草稿（Markdown / MDX）
3. 自我檢核（品牌語氣、事實正確、連結有效）
4. 提交 PR（參考 development-workflow.md）
5. 合併後自動或手動部署
6. 於社群或 Newsletter 推廣（Post-MVP）
```

## 6. 種子內容建議（MVP 啟動）

上線時建議至少具備以下種子內容：

| 類型 | 數量 | 建議主題 |
|------|------|----------|
| 部落格 | 2–3 篇 | 企業 AI 導入路線圖入門、Vibe Coding 簡介、AI Agent 工作流概覽 |
| 案例 | 1–2 則 | 匿名產業案例或方法論展示案例 |
| 課程 | 1–2 門 | Vibe Coding 工作坊、AI 工具企業培訓 |
| 模板 | 2–3 個 | AI 導入評估檢核表、Agent 工作流設計模板 |

## 7. SEO 與可發現性

- 每頁具備唯一的 `title` 與 `description`
- 文章使用語意化標題層級（H1 → H2 → H3）
- 內部連結：文章間互相引用，導向服務頁與聯絡頁
- URL 使用 kebab-case 英文 slug
- 圖片附 `alt` 文字

## 8. 內容品質檢核表

發布前確認：

- [ ] 標題與摘要清楚傳達價值
- [ ] 符合品牌語氣（專業、務實、可信賴）
- [ ] 技術內容經過驗證或標註為觀點
- [ ] 客戶資訊已匿名或取得授權
- [ ] frontmatter 欄位完整
- [ ] 連結與圖片可正常存取
- [ ] 有適當的 CTA（如「了解更多服務」「下載模板」）
