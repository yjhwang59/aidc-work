AI 輔助開發工具越來越強，但許多團隊的困境不是「不會用工具」，而是「用完之後不知道怎麼維護、怎麼協作、怎麼納入正式流程」。

本文分享一套在顧問專案中反覆驗證的工作流：**Cursor 負責日常開發與維護、Codex 處理結構化任務、GitHub 作為版本控管與協作中樞**。三者分工明確，才能從「個人加速」走向「團隊可複製」。

## 三者的角色定位

| 工具 | 主要用途 | 適合的任務類型 |
|------|----------|----------------|
| **Cursor** | IDE 內即時協作 | 功能開發、重構、除錯、程式碼審閱輔助 |
| **Codex** | 結構化指令執行 | 批次修改、測試修復、依規格產出程式碼 |
| **GitHub** | 版本控管與協作 | PR 審閱、Issue 追蹤、CI/CD、知識累積 |

關鍵原則：**AI 產出必須經過 Git 追蹤與 PR 審閱**，不能停留在本機對話紀錄裡。

## 建議工作流

```text
Issue / 需求
    ↓
Cursor：理解脈絡、規劃變更範圍
    ↓
Cursor / Codex：實作與測試
    ↓
本機驗證（build、lint、手動測試）
    ↓
Git commit → Push → 開 PR
    ↓
PR 審閱（人工 + AI 輔助審閱）
    ↓
合併 → 部署
```

### 階段一：需求進入

每個任務從 GitHub Issue 或明確的需求描述開始。Issue 中應包含：

- 要解決的問題（不是解法）
- 驗收條件
- 相關檔案或模組（若已知）
- 限制條件（如不引入新依賴、不改 API）

這讓 Cursor Agent 有足夠脈絡，避免「做對了但做錯題」。

### 階段二：Cursor 開發

在 Cursor 中開發時，建議遵循：

1. **先讀再寫**：讓 Agent 先閱讀相關檔案與專案慣例，再動手修改。
2. **最小變更**：一次 PR 聚焦一個主題，避免巨型 diff。
3. **保留對話紀錄**：重要決策寫進 commit message 或 PR 說明，而非只留在聊天視窗。

對於重複性維護任務，可搭配專案內的 [Cursor 維護任務提示詞範本](/templates/cursor-maintenance-task-prompt-template)，確保每次指令品質一致。

### 階段三：Codex 批次處理

當任務具備明確規格、可獨立驗收時，Codex 特別適合：

- 修復一組相關的 lint / type 錯誤
- 依命名慣例批次重構
- 為多個元件補齊相同模式的測試

使用 Codex 時，務必在 PR 中說明「哪些檔案由 Codex 產出、人工審閱了哪些部分」。

### 階段四：GitHub PR 與審閱

PR 是這套工作流的品質關卡：

- **標題與說明**：說明「為什麼改」而不只是「改了什麼」
- **測試計畫**：列出驗證步驟，方便審閱者複查
- **CI 必須通過**：build、lint、test 全綠才合併

審閱者可用 Cursor 協助理解 diff，但**合併決策仍由人負責**。

## 常見陷阱與對策

### 陷阱一：AI 改太多檔案

**對策**：在提示詞中明確限制範圍，例如「只修改 `components/` 下的檔案，不動設定檔」。

### 陷阱二：不跑 build 就提交

**對策**：將 `npm run build` 納入提交前檢查清單，必要時用 pre-commit hook 輔助。

### 陷阱三：每個人用不同的提示詞風格

**對策**：在 repo 中維護 `AGENTS.md` 或提示詞範本，讓團隊共用同一套指令規範。

## 團隊導入建議

若要在團隊中推廣這套工作流，建議分三階段：

1. **個人試點**（1–2 週）：1–2 位成員用 Cursor 完成小型任務並走完整 PR 流程。
2. **小組共識**（1 週）：整理提示詞範本、PR 規範、常見地雷清單。
3. **正式納入**（持續）：新功能與維護任務預設走 AI 輔助 + PR 審閱流程。

## 小結

Cursor、Codex、GitHub 各自擅長不同環節，串起來才是一套完整工作流。工具能加速寫程式，但**流程設計決定能否規模化**。

若你的團隊正在評估 AI 輔助開發導入，可從一個小型維護任務開始，搭配提示詞範本與 PR 規範，先建立可複製的節奏。
