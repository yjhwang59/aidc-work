> **案例說明**：本案例為顧問自有專案的方法論展示，非客戶委託案件。文中不涉及任何真實客戶名稱、聯絡資訊或商業機密。

## 背景

某企業 AI 發展顧問需要建立一個專業網站，作為長期累積的內容資產庫——包含文章、案例、模板與課程說明。目標不是做出華麗的展示頁，而是：

- 能持續發布顧問內容
- 架構簡單、維護成本低
- 展示 AI 輔助開發的實際工作流
- MVP 階段不引入資料庫或 CMS

## 面臨的挑戰

| 挑戰 | 說明 |
|------|------|
| 一人團隊 | 顧問需同時負責內容、開發與維運 |
| 內容與程式分離 | 文章、案例、模板需獨立於元件程式碼管理 |
| 快速迭代 | 希望能在短時間內完成可上線的 MVP |
| 可複製性 | 網站本身要成為「AI 輔助開發」的方法展示 |

## 解法

### 技術選型

採用 **Next.js App Router + TypeScript + Tailwind CSS** 的靜態優先架構：

- 內容以 Markdown 檔案存放於 `content/` 目錄
- 無資料庫、無 CMS、無身份驗證
- 靜態生成（SSG）確保部署簡單、效能穩定

### AI 輔助開發工作流

開發過程全程使用 Cursor 與 Codex，搭配 GitHub 進行版本控管：

```text
docs/ 規格文件 → Cursor Agent 理解專案脈絡
       ↓
元件與頁面開發（Cursor 即時協作）
       ↓
內容檔案批次產出（Codex 結構化任務）
       ↓
npm run build 驗證
       ↓
Git PR → 審閱 → 合併 → 部署
```

### 關鍵設計決策

1. **內容目錄結構**：`content/blog`、`content/cases`、`content/courses`、`content/templates` 四類，每篇含 frontmatter 中繼資料。
2. **文件驅動開發**：先建立 `AGENTS.md`、`docs/` 規格文件，讓 AI Agent 每次任務都有一致上下文。
3. **提示詞範本化**：維護任務使用標準化提示詞，確保 AI 產出品質穩定。
4. **最小依賴原則**：不引入非必要的 UI 函式庫或後端服務。

## 成果

| 指標 | 結果 |
|------|------|
| MVP 上線時間 | 數日內完成可部署版本 |
| 內容類型 | 支援文章、案例、課程、模板四類 |
| 維護成本 | 新增內容只需新增 Markdown 檔案 |
| 技術債 | 低——靜態架構、無外部服務依賴 |

質化成效：

- 網站本身成為 Vibe Coding 方法的活案例
- 文件與提示詞範本可複用到其他顧問專案
- 內容發布流程與程式開發流程統一走 Git PR

## 可複製的經驗

其他團隊若要採用類似做法，可參考以下要點：

### 1. 先寫文件，再寫程式

在讓 AI 動手之前，先準備：

- 專案目標與技術限制（如 `AGENTS.md`）
- 網站地圖與內容策略
- 品牌語氣與撰寫規範

AI 的產出品質，很大程度取決於上下文是否完整。

### 2. 內容與呈現分離

不要把長文硬編碼在 React 元件中。Markdown + frontmatter 讓非工程背景成員也能參與內容維護。

### 3. 每個 AI 產出都要可追溯

所有變更經 Git 追蹤，PR 說明記錄「為什麼改」。這對一人團隊尤其重要——三個月後你會感謝現在的自己。

### 4. 從小處驗證，再擴大範圍

先完成首頁與一個內容類型的列表/詳情頁，確認工作流順暢後，再批次產出種子內容。

## 下一步

此專案持續演進中，後續計畫包含：

- 從 `content/` 目錄動態讀取並渲染內容
- 標籤篩選與站內搜尋
- 更多種子內容與模板
